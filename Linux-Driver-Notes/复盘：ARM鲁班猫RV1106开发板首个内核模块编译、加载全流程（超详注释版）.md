## ## 复盘：在ARM开发板上加载第一个内核模块

**目标**：在鲁班猫RV06开发板上，编译并加载 `hello_drv.ko`，在内核日志中看到 `printk` 的输出。

**最终结果**：模块编译成功，加载成功（`lsmod` 可见），但因RNDIS调试信息刷屏，内核日志被淹没，未能直接看到输出。改用USB转TTL串口控制台即可解决。

### ### 第一阶段：开发板初始连接

**问题1：板子只有一个Type-C口，供电和RNDIS冲突**

**说白了就是就是支持把typec当成网线用的协议**

- 板子只有一个Type-C口，用来供电就不能同时用RNDIS

- **注释**：RV1106这类嵌入式开发板设计精简，仅单Type-C复用接口，硬件层面无法同时分开工况：单纯供电、数据传输、RNDIS虚拟网卡功能，默认只能二选一，是硬件设计特性而非故障。

- **解决**：用Type-C数据线连接电脑USB口，同时供电和走RNDIS协议。前提是线必须支持数据传输，不能是充电线。

- **注释**：市面上绝大多数充电线仅有供电铜线，无数据传输铜线，无法实现RNDIS网络协议、数据拷贝功能；仅原装/专用数据线支持数据+供电双复用，可让单Type-C口同时完成供电、联网两大功能。

**问题2：电脑无法识别RNDIS设备**

- 板子 `/etc/usb_config`里 `OTG_MODE=host`，导致USB口工作在主机模式，没有把自己模拟成网卡

- **注释**：USB OTG模式分为两种：host（主机模式，类似电脑USB口，用来接U盘、外设）、peripheral（从机模式，类似U盘、网卡，被电脑识别）。主机模式下板子是“主控”，无法被电脑识别为虚拟网卡，自然无法启用RNDIS。

- **解决**：需要改成 `OTG_MODE=peripheral`

- **注释**：改为从机模式后，板子主动模拟成USB网络设备，电脑系统可加载RNDIS驱动，识别出虚拟网卡，实现电脑与开发板局域网互通。

**问题3：无法修改TF卡上的配置文件**

- Windows不能原生读写ext4分区

- **注释**：Linux系统的系统分区、文件分区默认使用ext4格式，Windows系统仅原生支持NTFS、FAT32格式，无ext4读写驱动，直接插入TF卡无法查看、修改ext4分区内的文件。

- DiskGenius能找到文件但无法删除（收费限制）

- 虚拟机Ubuntu挂载TF卡失败（分区表无法识别）

- **注释**：TF卡包含多个分区（启动分区、系统分区、数据分区），分区表复杂，虚拟机USB直通、挂载识别容易出错，导致无法正常读取文件。

- `dd` + `sed` 无法直接编辑块设备

- **注释**：TF卡属于块设备，不是普通文本文件，无法直接用文本编辑命令修改内部配置，强行操作会报错甚至损坏分区数据。

- `parted` 报"无法辨识的磁盘卷标"

- **最终解决**：放弃在电脑上修改TF卡。第二天借到USB转TTL串口线，通过串口直接登录板子，`vi /etc/usb_config` 改掉配置。

- **注释**：串口是嵌入式开发板的底层万能调试入口，无需网络、无需挂载磁盘，只要板子正常启动，即可直接登录系统修改配置，规避所有磁盘读写、系统兼容问题。

### ### 第二阶段：建立网络连接

**问题4：改完USB配置后，电脑仍无RNDIS网卡**

- RNDIS驱动未安装或未正确识别

- **注释**：Windows系统不会自动适配嵌入式设备的RNDIS虚拟网卡驱动，首次连接大概率识别异常，需要手动干预配置。

- **解决**：设备管理器里出现"Remote NDIS based Internet Sharing Device"，手动配置IP（电脑端 `192.168.137.1`，板子端 `192.168.137.100`）。

- **注释**：手动配置同网段静态IP，可跳过DHCP自动获取失败的问题，直接实现电脑与开发板点对点局域网互通，是嵌入式RNDIS联网的通用解决方案。

**问题5：SSH登录密码错误**

- 输入了错误密码

- **注释**：串口登录密码、SSH登录密码为同一套系统密码，输错会直接拒绝连接，无其他报错提示，容易误导排查方向。

- **解决**：确认串口登录时使用的密码，重新输入。

**问题6：板子无法ping通外网**

- 板子能ping通电脑（`192.168.137.1`），但ping不通外网（`114.114.114.114`）

- **注释**：能ping通电脑代表局域网互通正常，无法ping外网是Windows网络共享、网关转发配置异常，属于上层网络配置问题，和板子硬件、驱动无关。

- Windows的网络共享未正确生效

- 尝试了 `dhclient usb0`，但板子上没装这个命令

- **结果**：外网未通，但不影响后续操作。文件通过MobaXterm的SFTP直接拖拽传输，不依赖板子联网。

- **注释**：内核模块编译、加载、调试全程不需要开发板联网，仅需本地文件传输和本地运行，SFTP局域网传输完全可以替代外网下载功能，不影响核心开发流程。

**问题9：SDK自带工具链缺失**

- `find` 找不到 `arm-linux-gnueabihf-gcc`

- **注释**：部分SDK源码仅提供内核代码，未预装对应版本的ARM交叉编译工具链，系统无法识别ARM架构编译命令，导致编译失败。

- **超详补充注释（全套工具链底层原理 + 工程疑问全解）**

- **1. 什么是编译工具链？**编译工具链不是单个命令，是**一整套嵌入式编译必备工具集合**，属于成套工具包。核心包含：gcc（C编译器）、g++（C++编译器）、ld（链接器）、as（汇编器）、objcopy（格式转换）、strip（瘦身）、ar（静态库打包）等全套工具，缺一不可，共同完成源码 → 汇编 → 链接 → 可执行文件/内核模块的完整编译流程。

- **2. 工具链有什么用？为什么编译必须要它？**裸源码只是文本，CPU无法识别，必须依靠工具链：语法检查、编译翻译、代码链接、内存重定位、架构适配，最终生成对应硬件架构的机器码。没有工具链，内核、驱动、程序永远无法运行，源码只是无用文本文件。

- **3. 什么是「SDK对应的ARM原厂交叉编译工具链」？**x86电脑编译出来的程序只能在电脑跑，不能在ARM开发板跑。原厂SDK配套工具链是**专门针对RV1106架构、内核版本、系统库、指令集（Thumb2/ARMv7）定制编译好的专属工具包**。和通用ARM工具链不同，它和板子内核的版本魔术、编译配置、ABI接口、系统库100%匹配，是模块能正常加载、内核能正常编译的硬性前提。

- **4. 能不能用 apt 一个个下载、手动摆放文件凑出工具链？**绝对不行。原因：apt安装的是通用版ARM工具链，版本、编译参数、库文件、ABI标准和原厂SDK不匹配；且工具链内部各工具存在**版本强绑定依赖**，手动拼凑会出现链接不通过、符号不匹配、版本魔术错乱、模块insmod报错，属于嵌入式开发大忌，100%翻车。

- **5. 为什么工具链必须解压到 /opt 目录？/opt 是什么意思？**/opt 是 Linux 系统**第三方软件/商业工具/自定义工具的标准安装目录**，全称 optional（可选软件目录）。系统约定：系统自带软件放 /usr，用户手动安装的外部工具、开发工具统一放 /opt，干净、不污染系统目录、全局可管理、适配所有用户权限，是嵌入式工具链的标准存放路径。

- **6. 为什么必须设置 PATH 环境变量？**Linux终端只会在 PATH 记录的目录里检索命令。工具链解压在 /opt 自定义路径，系统默认不认识这些 arm-xxx 命令。写入 PATH 后，全局任意目录都能识别交叉编译命令，否则只能进工具链bin目录才能执行编译，无法配合内核Makefile自动化编译。

- **核心工程结论（对应你版本魔术报错）**：工具链不匹配 = 内核配置校验不通过 = version magic 错乱 = 模块格式报错，这也是新手90%模块加载失败的根源。

- **超详补充：版本魔术、ARMv7、Thumb2、工具链、PATH 终极通俗答疑（修正+固化正确认知）**

- **1. 什么是内核 Version Magic（版本魔术）？**不是版本号，是内核编译完成后生成的**唯一校验指纹**。由：内核版本号、SMP多核开关、Thumb2指令集开关、内存模型、编译架构、内核配置项共同哈希计算生成。作用：Linux内核强制校验，**模块编译时的指纹必须和板子运行内核的指纹完全一致**，差一个配置选项都会指纹不匹配、insmod报错 invalid module format。你本次的报错就是典型：内核开启Thumb2、关闭SMP，编译工具链环境开启SMP、关闭Thumb2，指纹错乱。

- **超详专项补充（你本次手动 FLAG 配置深度答疑）**

- **问题1：是不是工具链、内核源码、模块编译的 SMP、Thumb2、ARMv7 开关必须完全一致，模块才能加载成功？答案：是，必须 100% 一致。**版本魔术是全局哈希指纹，只要板子运行内核和编译模块的「核心架构配置」有一项不同，指纹直接改变。关键匹配项：SMP（多核开关）、Thumb2（精简指令集）、ARM架构版本、内存分页模型。这也是为什么你内核版本号都是 5.10.160，依然加载失败的核心原因：**版本号相同 ≠ 编译配置相同**。

- **问题2：你之前写的 EXTRA_CFLAGS 到底做了什么？**`EXTRA_CFLAGS += -mthumb -D__thumb2__ -D__LINUX_ARM_ARCH__=7`，**你的理解方向对，但作用层级完全错了**。

- **精准拆解：**这行代码的作用是：**临时修改当前驱动模块的编译参数**，强制让这个 .c 文件按照 Thumb2、ARMv7 编译。**但它绝对改不了内核全局配置、改不了版本魔术、改不了 SMP 开关**。

- **误区纠正（非常重要）**：

- 1. **SMP 属于内核全局宏**：由顶层 .config / CONFIG_SMP 控制，**模块 CFLAGS 无法覆盖**，你加编译参数也修不掉 SMP 差异；

- 2. 你手动加 `-mthumb -D__thumb2__` 只是「让驱动代码用 thumb2 指令编译」，但**内核整体的版本魔术指纹早已固定**，不会因为单个模块编译参数改变而刷新；

- 3. 所以你当时加这行代码：**治标不治本，版本魔术依旧不匹配，insmod 照样报错。**

- **【唯一正确、可落地的终极解决办法】**

- 1. 进入内核源码顶层目录，修改内核全局配置文件 `.config`，让编译配置 **完全对齐开发板出厂内核配置**：关闭多余的`CONFIG_SMP` 多核功能、开启 `CONFIG_THUMB2_KERNEL` 精简指令集，保证架构参数和板子运行内核一模一样；

- 2. **关键必做步骤**：修改完 .config 后，必须重新执行 `make modules_prepare`，刷新内核所有编译头文件、依赖规则、版本魔术指纹，让编译环境完全同步新配置；

- 3. 重新进入驱动目录执行 make 编译模块，此时模块的 version magic 会和开发板内核完全匹配，彻底解决 `invalid module format` 报错。

- **核心结论**：版本魔术匹配看的是「内核全局编译环境」，不是单个驱动文件的局部编译参数，只能改内核全局配置+刷新编译环境，无法靠驱动CFLAGS强行兼容。

- **最终正确逻辑（你本次踩坑总结的真理）**

- 想要版本魔术一致，只能：**修改内核顶层 .config 全局配置 → 重新 modules_prepare 刷新内核编译环境 → 再编译模块**。

- 绝对不能靠模块局部 CFLAGS 去硬凑匹配，局部参数无法修改内核全局指纹。

- **2. ARMv7、Thumb2 是什么？是工具命令吗？不是编译命令、不属于工具链命令**，二者是**ARM硬件CPU的指令集架构规则**，是CPU能听懂的语言标准：① ARMv7：是你RV1106开发板CPU的**架构大版本**（32位ARM经典架构），规定了CPU通用指令、运行规则；② Thumb2：是 ARMv7 架构下的**压缩精简指令集**，特点是指令更短、执行效率更高、节省功耗，嵌入式ARM板子基本全开。简单理解：ARMv7是「语言大类」，Thumb2是「语言精简方言」，属于硬件架构属性，和gcc、ld等编译命令无关。

- **3. 你对编译工具链的理解：完全正确**。编译工具链本质就是**跨架构编译器合集**，和PC端C语言gcc编译器原理一致。核心工作：把人类能看懂的文本源码（.c），经过预处理、编译、汇编、链接，翻译成ARM CPU能听懂的二进制机器码。普通gcc只能编译x86电脑机器码，而原厂ARM工具链专门编译 RV1106 ARMv7 架构机器码。

- **4. 你对PATH环境变量的理解：大体正确，精准补全定义**。PATH是系统的**命令检索目录列表**，不是存放命令文件，是**记录命令所在文件夹路径**。终端输入命令时，会按PATH顺序遍历目录查找对应程序。把工具链bin目录写入PATH，就是告诉系统：全局任何位置都能识别 arm-xxx 交叉编译命令，无需输入绝对路径执行，适配内核Makefile自动编译。

**问题10：内核源码未配置，无法编译模块**

- **【终极答疑：彻底区分 .config顶层配置 / modules_prepare / 驱动make 三者关系（解决你所有疑惑）】**

- **1. 你改的顶层 .config 里的 ARMv7、Thumb2、SMP 是什么？**

- 这是**内核本体的全局硬件架构开关**。

- 它决定：你这一套内核源码「整体是什么架构、支持什么指令集、是否多核」。

- **2. make defconfig 作用是什么？**

- 清空旧配置，生成一套完整默认内核全局配置文件 `.config`，包含架构、指令集、SMP、缓存、模块规则等所有内核参数。

- **3. make modules_prepare 到底是干嘛的？执行节点在哪里？**

- 它是 **内核源码层面的「模块编译环境初始化命令」**，专属前置命令，和你驱动文件夹的 make 完全不是一回事。

- **执行时机（唯一正确时序）：**

- ① 修改顶层 .config 改好架构/指令集/SMP → ② **必须执行 make modules_prepare** → ③ 再编译驱动ko

- **核心作用：**

- 根据最新 .config 刷新：内核头文件、编译脚本、版本魔术指纹、模块校验规则、符号表。

- **关键真相：你改了 .config 不执行这条命令 = 配置完全不生效**，内核依旧沿用旧指纹，模块永远不匹配。

- **4. 它和你驱动目录下的 make 是同一个东西吗？【重点区分】**

- **完全不是！层级完全不同**

- **① make modules_prepare**：在内核源码顶层目录执行，**初始化整个内核的模块编译环境**，属于「内核级准备」。

- **② 你驱动文件夹的 make**：调用内核已经准备好的编译体系，**编译你自己的 hello_drv.c**，属于「用户驱动编译」。

- **层级关系**：先有内核环境准备，才有驱动编译。

- **5. 终端直接打 make 和 Makefile 里的 modules 是什么关系？**

- 你驱动目录的 Makefile 中：

- 终端输入 make，本质就是执行 Makefile 里的 modules 规则，自动调用内核源码的编译体系。

- 但是！**如果内核没执行过 modules_prepare**，内核编译体系残缺、指纹老旧，你怎么 make 都是错的、版本不匹配。

- **6. 一句话总结完整闭环（你本次踩坑的真理）**

- .config 决定内核真实架构身份（指纹），modules_prepare 刷新生效，驱动 make 只是借用环境编译模块。

- `export ARCH=arm`

- `export CROSS_COMPILE=arm-rockchip830-linux-uclibcgnueabihf-`

**`insmodinvalid module format`** **问题14： 报 **

- `dmesg` 显示 `version magic mismatch`

- **注释**：version magic（版本魔术）是内核专属校验码，由内核所有配置选项共同生成，用于校验模块和运行内核是否同源、同配置，不一致直接拒绝加载模块。

### ### 第五阶段：加载成功但日志不可见

**`insmodlsmodhello_drvdmesg`** **问题15： 无报错， 能看到 ，但  看不到日志**

- 模块加载成功（`lsmod` 可见，`[permanent]` 标记）

- **注释**：lsmod能查到模块，代表模块已经成功载入内核内存、注册完成，驱动本身编译、加载完全无问题，不属于开发失败。

- `printk` 输出被RNDIS调试信息淹没（RNDIS每几毫秒刷好几条日志，内核日志环缓冲区被冲掉）

- **注释**：内核日志是环形缓冲区，存储空间有限，新日志会覆盖旧日志；RNDIS虚拟网卡驱动高频打印调试日志，快速填满缓冲区，直接覆盖了模块初始化的printk日志。

- SSH终端看不到内核日志（控制台日志级别过滤）

- `[permanent]` 标记导致 `rmmod -f` 也无法卸载

- **注释**：permanent代表模块被内核永久固化加载，属于内核常驻模块，无法通过rmmod卸载，只能重启开发板清除。

**尝试的解决方案**：

1. 把 `KERN_INFO` 改成 `KERN_EMERG` → 仍被淹没

2. **注释**：最高紧急日志级别也无法对抗高频刷屏日志，缓冲区覆盖是硬件级缓存机制，和日志级别无关。

3. 用 `kernel_write` 直接写 `/dev/kmsg` → 仍被淹没

4. 调整`/proc/sys/kernel/printk` 日志级别 → 无效，RNDIS刷屏太快

5. 关闭RNDIS调试输出 → 未成功

**根本原因**：通过SSH终端操作时，内核日志不会主动推送到SSH会话。而RNDIS驱动的大量调试信息迅速填满了日志缓冲区。

**最终方案**：使用USB转TTL串口线连接板子的调试串口。串口终端是内核控制台的直接输出通道，所有 `printk` 输出都会实时显示，不受SSH和日志缓冲区的限制。

**注释**：串口是Linux内核原生控制台，系统启动、内核打印、驱动日志、内核崩溃信息都会实时直出，不会被缓存、过滤、覆盖，是嵌入式驱动调试的唯一可靠通道。

1. **串口线是嵌入式开发的最后保障**：网络不通、SSH连不上、内核崩溃时，只有串口能看到系统状态。

### ### 专项补充：彻底搞懂 RNDIS 协议（逐问拆解）

**1. RNDIS 全称是什么？**

**RNDIS = Remote Network Driver Interface Specification**

中文：**远程网络驱动接口规范**，是微软主导制定、适配Windows系统的USB网络通信专属协议，是嵌入式设备与Windows电脑局域网互通的核心标准。

**2. RNDIS 到底有什么用？**

一句话核心定义：**RNDIS 可以让一根普通 USB 数据线，模拟出一张「虚拟网卡」**。

常规设备上网需要网线、以太网口，而RV1106开发板、手机这类小型设备没有网口，就靠 RNDIS 实现：**USB 传输 TCP/IP 网络数据包**，让电脑和开发板直接局域网互通，实现 SSH 连接、文件传输、设备调试等功能。

**3. RNDIS 是用来实现供电和传数据的协议吗？**

**不是！完全区分开**（新手最大误区）：

- **供电**：是 USB 硬件本身的物理能力，和任何协议无关，插上数据线就供电，无需任何配置。

- **普通数据传输**（拷贝文件）：是 USB Mass Storage、MTP 协议负责。

- **RNDIS 只干一件事**：**让 USB 跑网络、跑网卡数据流**，专门负责局域网通信，不负责供电、不负责普通文件拷贝。

所以你板子 Type‑C 一线复用逻辑是：**USB硬件供电 + RNDIS协议跑网络**，两者独立并行、互不干扰。

**4. 和「手机插电脑传文件」有什么区别？（核心对比）**

很多人混淆，我直接对标讲透：

**① 手机插电脑默认模式：MTP/PTP 协议**

- 用途：**单纯拷贝照片、视频、文件**

- 本质：存储设备数据传输

- 特点：只能传文件，**没有网络、没有IP、不能SSH、不能ping**

**② 开发板 RNDIS 模式：RNDIS 网络协议**

- 用途：**建立双向局域网通道**

- 本质：TCP/IP 网络通信（USB模拟网卡）

- 特点：有独立IP、可ping通、可SSH登录、可SFTP传文件、可调试网络功能

**5. 通俗终极区别**

- **普通手机数据线连接** = 只是「U盘模式」，只能搬文件。

- **RNDIS 连接** = 相当于「插了一根网线」，设备和电脑组成小型局域网，支持所有网络操作。

**6. 对应你本次踩坑的原理复盘**

你之前 `OTG_MODE=host` 失败的根本原因：

- Host（主机模式）：板子当电脑，只能接U盘、外设，**不能模拟网卡**，RNDIS 失效。

- Peripheral（从机模式）：板子当外设，主动启动 RNDIS 协议，被电脑识别为虚拟网卡，网络正常打通。

**7. 解释日志刷屏问题根源**

RNDIS 为了维持网络链路保活、校验数据包，会**毫秒级高频收发心跳包、校验包**，持续输出内核调试日志，快速占满内核环形缓冲区，直接覆盖你驱动加载的单次 printk 输出，这也是你模块加载成功但看不到日志的核心协议层面原因。

### ### 个人全程核心总结（最终定稿、误区全修正）

1. 关于RNDIS协议：RNDIS是专属USB转网口协议，核心作用是让开发板Type-C数据线替代物理网线，实现电脑与开发板局域网互通、SSH连接、SFTP文件传输。但RNDIS会高频发送保活心跳包，产生**日志风暴**，快速填满内核环形缓冲区，直接覆盖驱动初始化的printk日志，导致模块加载成功却看不到打印信息，唯有串口控制台能规避该问题、稳定查看内核日志。

2. 关于Git裸仓库：SDK解压后仅含.git文件夹，属于裸仓库，无完整可编译源码。必须使用本地克隆命令 `git clone 裸仓库路径 自定义文件夹名`，才能还原出完整的内核源码、编译脚本和配置文件，正常开展编译工作。

3. 关于交叉编译工具链：原厂配套ARM交叉编译工具链，需解压在Linux `/opt` 目录，该目录是系统专属存放**第三方、外部自定义工具**的标准路径，不污染系统原生目录。同时必须将工具链bin目录添加到PATH环境变量，终端才能全局识别arm架构编译、链接命令，完成跨架构编译。临时export配置仅当前终端生效，新开终端会失效，需写入.bashrc实现永久生效。

4. 关于版本魔术与编译配置（**重点修正模糊认知**）：编译模块通过版本魔术校验的核心，**不是把内核配置改成和工具链一致**，完全相反！正确逻辑是：工具链是固定适配板子出厂内核的，必须手动修改内核顶层 `.config`，关闭SMP、开启Thumb2，**让内核源码配置100%对齐开发板正在运行的出厂内核配置**，再执行 `make modules_prepare` 刷新编译环境、更新版本魔术指纹，编译出的ko模块才能通过内核校验，避免格式不匹配报错。工具链版本固定，无法反向适配错乱的内核配置。

**新增深度答疑：出厂内核配置与硬件寄存器、可改性、技术范畴（彻底理清层级）**

**1. 出厂内核配置 ≠ 硬件寄存器配置（核心区分）**
