# Makefile copy伪目标 完整原理说明

问题：关于copy伪目标，将内核模块加载到开发板，这句话怎么理解，开发板会读取那个copy的文件夹里面的.ko文件吗？



## 一、先解答你的核心疑问（直白结论）

**会！开发板就是读取这个copy文件夹里的 .ko 文件来加载模块**

你写的这条命令：

```makefile
copy:
	sudo cp *.ko /home/embedfire/workdir
```

核心逻辑：**把电脑编译好的 .ko 文件，自动丢进开发板能访问的共享目录**，省去手动传输文件的步骤。

## 二、这个 /home/embedfire/workdir 是什么文件夹？

这是 **NFS 共享目录**（嵌入式开发标配）：

- 电脑（Ubuntu）上的 `/home/embedfire/workdir`

- 会被**挂载到开发板的系统中**

- 开发板可以直接访问、读取、执行这个文件夹里的所有文件

简单理解：**电脑和开发板共用的一个公共文件夹**。

## 三、copy 伪目标的完整工作流程

1. 你在电脑执行 `make`，编译出 `helloworld.ko`

2. 你执行 `make copy`

3. 系统自动把当前目录的 `*.ko` 全部复制到 **NFS共享目录 /home/embedfire/workdir**

4. 开发板端能实时看到这个 .ko 文件

5. 开发板直接执行 `insmod helloworld.ko` 加载模块

## 四、关键问题精准回答

### 1. 开发板会不会读取这个文件夹的 .ko？

**完全正确，会读取**。

开发板没有磁盘、没有编译器，不能自己编译 .ko，只能读取电脑共享过来的 .ko 文件加载运行。

### 2. 为什么不直接在当前目录加载？

因为：**当前目录是电脑本地目录，开发板看不见**。

只有放进 **NFS共享目录**，开发板才能跨设备读取文件。

### 3. copy 伪目标的本质作用

不用你手动 `cp`、不用手动 `scp` 传到开发板，一键自动部署模块，属于**开发便捷工具**，不属于编译必需步骤。

## 五、极简总结（可直接背诵）

copy伪目标的作用是将电脑编译生成的内核模块.ko文件，自动复制到电脑与开发板互通的NFS共享文件夹中，开发板可以直接读取该文件夹下的.ko文件，执行insmod命令完成内核模块加载，实现模块一键部署，无需手动传输文件。

2. **开发便利**：copy（自定义伪目标，不写也能正常编译，只是需要手动传文件）
