# read(2)                                                                系统调用手册                                                                read(2)

# 名称（NAME）

read - 从文件描述符读取数据

# 库（LIBRARY）

标准C库（libc, -lc）

# 语法（SYNOPSIS）

#include <unistd.h>

ssize_t read(int fd, void buf[.count], size_t count);

# 描述（DESCRIPTION）

read() 尝试从文件描述符 fd 中读取最多 count 个字节的数据，存入以 buf 为起始地址的缓冲区中。

对于支持定位（seeking）的文件，读取操作从文件偏移量（file offset）处开始，且文件偏移量会根据读取的字节数进行递增。如果文件偏移量已达到或超过文件末尾，则不会读取任何字节，read() 返回 0。

如果 count 为 0，read() 可能会检测到下文描述的错误。在没有任何错误的情况下，或者如果 read() 不检查错误，count 为 0 的 read() 会返回 0 且无其他任何影响。

根据 POSIX.1 标准，如果 count 大于 SSIZE_MAX，结果由具体实现定义；详见 NOTES 部分关于 Linux 系统的上限说明。

# 返回值（RETURN VALUE）

成功时，返回读取到的字节数（返回 0 表示文件末尾），且文件位置会相应前进该字节数。返回的字节数小于请求的字节数并非错误；例如，可能是因为当前实际可用的字节数较少（可能是因为接近文件末尾、从管道读取或从终端读取），或者 read() 被信号中断。另见 NOTES 部分。

失败时，返回 -1，并设置 errno 以指示错误类型。在这种情况下，文件位置（如果存在）是否改变未作明确规定。

# 错误（ERRORS）

EAGAIN：文件描述符 fd 指向非套接字文件，且已被标记为非阻塞（O_NONBLOCK），此时读取操作会阻塞。关于 O_NONBLOCK 标志的更多细节，参见 open(2)。

EAGAIN 或 EWOULDBLOCK：文件描述符 fd 指向套接字，且已被标记为非阻塞（O_NONBLOCK），此时读取操作会阻塞。POSIX.1-2001 允许返回这两种错误中的任意一种，且不要求这两个常量值相同，因此可移植的应用程序应同时检查这两种情况。

EBADF：fd 不是有效的文件描述符，或未以读取模式打开。

EFAULT：buf 位于进程不可访问的地址空间之外。

EINTR：在读取到任何数据之前，调用被信号中断；参见 signal(7)。

EINVAL：fd 关联的对象不适合读取；或者文件以 O_DIRECT 标志打开，且 buf 指定的地址、count 指定的值或文件偏移量未正确对齐。

EINVAL：fd 是通过调用 timerfd_create(2) 创建的，且向 read() 提供了错误大小的缓冲区；更多信息参见 timerfd_create(2)。

EIO：I/O 错误。例如，当进程处于后台进程组，试图从其控制终端读取数据，且该进程正在忽略或阻塞 SIGTTIN 信号，或者其进程组已成为孤儿进程组时，会出现此错误。当从磁盘或磁带读取时发生底层 I/O 错误，也可能出现此错误。在网络文件系统上，EIO 还可能因文件描述符上的建议锁（advisory lock）已被持有且该锁已丢失而产生。更多细节参见 fcntl(2) 的 Lost locks 部分。

EISDIR：fd 指向一个目录。

根据 fd 所关联的对象不同，还可能出现其他错误。

# 注意（NOTES）

在 Linux 系统上，read()（及类似系统调用）最多会传输 0x7ffff000（2,147,479,552）个字节，并返回实际传输的字节数。（32 位和 64 位系统均如此。）

在 NFS 文件系统上，读取少量数据时，仅第一次会更新时间戳，后续调用可能不会更新。这是由客户端属性缓存导致的，因为大多数（如果不是全部）NFS 客户端会将 st_atime（最后文件访问时间）的更新交给服务器处理，而从客户端缓存中满足的读取请求不会导致服务器端的 st_atime 更新（因为没有服务器端的读取操作）。可以通过禁用客户端属性缓存来实现 UNIX 语义，但在大多数情况下，这会显著增加服务器负载并降低性能。

# 缺陷（BUGS）

根据 POSIX.1-2008/SUSv4 第 XSI 2.9.7 节（“线程与常规文件操作的交互”）：

> 当以下所有函数操作常规文件或符号链接时，它们在 POSIX.1-2008 规定的影响范围内应彼此原子化：……
> 
> 

后续列出的 API 包括 read() 和 readv(2)。而线程（及进程）间应原子化的影响包括文件偏移量的更新。然而，在 Linux 3.14 之前，情况并非如此：如果两个共享同一个打开文件描述（参见 open(2)）的进程同时执行 read()（或 readv(2)），则 I/O 操作在更新文件偏移量方面并非原子化，导致两个进程中的读取操作可能（错误地）在它们获取的数据块中重叠。此问题已在 Linux 3.14 中修复。
