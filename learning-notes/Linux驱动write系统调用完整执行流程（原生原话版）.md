# Linux驱动write系统调用完整执行流程（原生原话版）

你写了一个hello驱动文件 `hello_drv.c`，编译成 `hello_drv.ko`，用 `insmod` 加载进内核。这个驱动里有一个回调函数 `hello_write`，它的作用很简单——收到数据就打印。你把它的地址填进 `file_operations` 结构体的 `.write` 字段里，告诉内核：“以后有人往我的设备写数据，你就调这个函数”。

然后你用 `mknod` 创建了 `/dev/hello` 这个**设备文件**。它不是存数据的普通文件，它是一个通往内核驱动的入口标记。主设备号把它和你的 `hello_drv` 绑在一起。

现在，你的应用程序调用 `write(fd, "hello", 5)`。这个 `fd` 来自 `open("/dev/hello")`。**这个 write 不是普通函数，是系统调用。** glibc 把它封装好，把系统调用号 4 放进寄存器，然后执行 `SVC` 指令。

CPU 执行 `SVC` 时，**硬件自动保存当前程序的现场**（程序计数器、状态寄存器），**然后把程序状态寄存器里的模式bit从0改成1，切换到内核态**。接着硬件跳到**异常向量表**的固定地址——这是CPU设计时焊死的，`SVC` 指令对应 `0x08` 偏移。那里存的是一条跳转指令，跳到内核的系统调用处理函数。

**你的应用程序到这里暂停了。** 它的状态被保存下来，等内核干完活再恢复。现在CPU开始执行内核代码。

内核入口代码从寄存器里读到系统调用号4，查**系统调用表**找到 `sys_write`。`sys_write` 调用 `vfs_write`。**VFS，虚拟文件系统，是一个分拣中心。** 它根据文件描述符找到 `/dev/hello` 对应的 `struct file`，从里面取出 `file_operations` 指针。

VFS 看到，这个文件的 `file_operations.write` 指向你之前注册的 `hello_write`。**这就是回调——你从来没自己调用过 hello_write，是内核通过VFS回头来调用你注册的函数。** 你的回调函数被触发。

在 `hello_write` 里面，第一件事不是打印。**用户空间传过来的 buf 指针，内核不能直接读。** CPU硬件禁止内核态直接碰用户空间的内存——这是安全隔离。所以必须用 `copy_from_user`，把用户空间的`"hello"` 从 `buf` 安全复制到内核空间的 `kbuf`。这一步做完，数据才真正“穿过”了那道墙。

然后 `printk` 打印，层层返回。内核入口恢复之前保存的现场，CPU切回用户态，你的应用程序从 `write()` 的下一行继续执行。你敲 `dmesg`，看到 `hello: received 'hello'`。

**从用户空间的 write，到内核驱动的 printk，数据穿过用户态→内核态边界，穿过VFS分拣，穿过回调函数指针，穿过 copy_from_user 安检窗口。你亲眼看到了全过程。**
