## 第一篇：CAN总线的“公告板”模型

```markdown
# CAN总线的“公告板”模型

## 一句话理解
CAN总线不是点对点通信，而是一块公告板。你往上贴公告（发送），别人自己决定看不看（接收）。不是发给特定的人，是别人根据CAN ID决定要不要听。

## 为什么这个模型重要
我刚开始学CAN的时候，习惯用串口的思维去理解——串口是两个人打电话，你一句我一句。CAN完全不是这样。CAN是一群人在看同一块公告板，每个人只关心自己感兴趣的消息。

## 模型拆解

### 1. 贴公告 = 发送CAN帧
- 你写好一个 `struct can_frame`，设置好 `can_id` 和数据
- 调用 `write(sock, &frame, sizeof(frame))`
- 这个帧被广播到总线上，所有节点都能看到
- 不是“我发给你”，是“我把消息贴在公告板上”

### 2. 看公告 = 接收CAN帧
- 每个节点都在监控总线
- 收到帧后，先看 `can_id`
- 如果这个ID是自己关心的，就处理数据
- 如果不关心，就忽略
- 不是“你为什么要发给我”，是“我选择看不看你的公告”

### 3. CAN ID = 公告分类标签
- CAN ID不是“发送者地址”，也不是“接收者地址”
- 它是“这条公告是什么类型”的标签
- 比如约定：
  - ID 0x100 = 温度数据
  - ID 0x101 = 湿度数据
  - ID 0x102 = 气压数据
- 所有节点根据ID决定“这条公告我需不需要处理”

## 我在网关里的实战应用
- **采集线程**：把Modbus传感器数据打包成CAN帧，贴到公告板上
- **上云线程**：监听公告板，看到自己关心的ID，取数据打包成JSON发MQTT
- **两个线程完全解耦**：采集线程不知道上云线程的存在，上云线程不知道采集线程是谁
- 采集线程只负责贴公告，上云线程只负责看公告

## 我踩过的坑
SocketCAN默认不接收自己发送的帧。因为公告板模型的逻辑是“别人看你的公告”，默认不让自己收到自己的。我在网关里采集线程和上云线程是同一个程序里的两个线程，需要通过公告板传递数据，所以要设置 `CAN_RAW_RECV_OWN_MSGS` 套接字选项才能自己收到自己的帧。

## 面试话术
> CAN总线我理解它的公告板模型——不是点对点通信，是节点往总线上贴公告，其他节点根据CAN ID决定要不要接收。这个模型让我在网关多线程设计时做了很好的解耦，采集线程和上云线程完全独立。
```

## 第二篇：MQTT的“邮局”模型

```markdown
# MQTT的“邮局”模型

## 一句话理解
MQTT Broker不是数据库，是邮局。它不存你的数据，它只做路由转发。你告诉它“我对什么话题感兴趣”（订阅），它就在有消息来的时候转给你。

## 为什么这个模型重要
我一开始以为Broker是个数据库，数据发上去存着，想看的时候去查。后来发现完全错了。Broker是实时的路由转发器，它不存历史数据，只负责“现在有人发消息了，赶紧告诉订阅这个话题的人”。

## 模型拆解

### 1. 订阅 = 告诉邮局“我住哪，我对什么感兴趣”
- 你连接Broker，订阅 `/sensor/temperature`
- Broker在自己的订阅表里记录：`/sensor/temperature` → [你的客户端ID]
- 这是“通知我”的意愿声明，不是“发给我”的命令

### 2. 发布 = 把消息送到邮局
- 你发布消息到 `/sensor/temperature`，内容是 `{"temp": 23.5}`
- Broker收到后，查订阅表：谁订阅了 `/sensor/temperature`？
- 找到所有订阅了这个Topic的客户端，把消息转发过去
- 你没指定发给谁，是Broker根据订阅表决定转给谁

### 3. Topic = 投递地址，不是IP地址
- Topic是逻辑上的“投递地址”，按 `/` 分层
- 比如 `/factory1/line2/sensor/temperature`
- 支持通配符：`/factory1/line2/#` 可以匹配下面所有子Topic
- 这不是IP地址，不需要知道接收者在哪台机器上

### 4. Broker = 订阅表 + 转发引擎
- Broker维护一张表：Topic → 订阅者列表
- 收到消息 → 查表 → 转发 → 结束
- 不存数据，不记历史，做完就忘
- 如果你想要历史数据，需要单独做数据存储，MQTT不管这个

## 我在网关里的实战应用
- 网关采集到传感器数据后，发布到华为云IoTDA的Topic
- 华为云IoTDA是Broker，收到后转发给订阅了这个Topic的云应用
- 网关发布时不需要知道云应用的IP地址，只需要知道Topic
- 断线时我用指数退避重连，因为MQTT连接是基于TCP的长连接

## 面试话术
> MQTT我理解它是邮局模型。Broker不是数据库，是路由转发器，维护订阅表和转发引擎。设备发布消息到Topic，Broker根据谁订阅了这个Topic决定转发给谁。这种模型天然适合物联网场景，因为设备不需要互相知道IP地址，只需要约定Topic就行。我的网关项目就是通过MQTT把传感器数据上报到华为云IoTDA。
```
