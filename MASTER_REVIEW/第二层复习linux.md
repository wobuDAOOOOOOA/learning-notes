字符驱动：
其实具体的代码我还记不得，我现在只记得是，再应用层程序调用 `write(fd, "hello", 5)`。这个 `fd` 来自 `open("/dev/hello")`。**这个 write 不是普通函数，是系统调用。** glibc 把它封装好，把系统调用号 4 放进寄存器，然后执行 `SVC` 指令。
执行的同时，硬件上cpu的程序状态寄存器的bit从0变成1，从用户态转变成内核态接着硬件跳到**异常向量表**的固定地址——这是CPU设计时焊死的，`SVC` 指令对应 `0x08` 偏移。那里存的是一条跳转指令，跳到内核的系统调用处理函数。应用程序到这里暂停了。它的状态被保存下来，等内核干完活再恢复。现在CPU开始执行内核代码。
内核入口代码从寄存器里读到系统调用号4，查**系统调用表**找到 `sys_write`。`sys_write` 调用 `vfs_write`。**VFS，虚拟文件系统，是一个分拣中心。** 它根据文件描述符找到 `/dev/hello` 对应的 `struct file`，从里面取出 `file_operations` 指针。
VFS 看到，这个文件的 `file_operations.write` 指向你之前注册的 `hello_write`。**这就是回调，从来没自己调用过 hello_write，是内核通过VFS回头来调用你注册的函数。** 回调函数被触发。
在 `hello_write` 里面，第一件事不是打印。**用户空间传过来的 buf 指针，内核不能直接读。** CPU硬件禁止内核态直接碰用户空间的内存——这是安全隔离。所以必须用 `copy_from_user`，把用户空间的`"hello"` 从 `buf` 安全复制到内核空间的 `kbuf`。这一步做完，数据才真正“穿过”了那道墙。
然后 `printk` 打印，层层返回。内核入口恢复之前保存的现场，CPU切回用户态，应用程序从 `write()` 的下一行继续执行。敲 `dmesg`，看到 `hello: received 'hello'`。

## pthread：修正踩坑细节

你记的对，踩坑点是**rtu和tcp采集都用到了my_can_send函数**。两个采集线程可能同时调用这个函数，导致CAN帧数据竞争——一个线程写到一半，另一个线程也进来写，数据就乱了。

**修正后的踩坑表述**：

> RTU采集和TCP采集两个线程都会调用CAN发送函数。如果不加锁，两个线程可能同时往CAN总线上写帧，导致数据竞争。我用互斥锁把CAN发送函数保护起来，同一时刻只有一个线程能发帧。

## candump：补充命令

你问的"什么命令查看有没有创建好虚拟接口"：

```bash
# 查看所有CAN接口
ip link show

# 只看vcan0
ip link show vcan0

# 如果不存在，创建虚拟CAN接口
sudo ip link add dev vcan0 type vcan
sudo ip link set up vcan0

# candump监听CAN总线数据
candump vcan0

# cansend发送测试帧
cansend vcan0 123#11223344
```

**面试话术**：

> 调试CAN自收发问题时，先用`ip link show vcan0`确认虚拟接口存在且状态UP。然后用`candump`监听总线，确认发送函数确实把帧发出去了。再用`cansend`手动发帧测试接收通路。两端都通，说明问题不在硬件和驱动层，最后查到是内核选项`CAN_RAW_RECV_OWN_MSGS`默认关闭。
